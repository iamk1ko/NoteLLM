# 后端核心业务设计与接口约定

说明：
- 本文档用于后端核心业务流程、接口约定与开发规范的统一说明
- 目标：方便前后端对接与新人快速理解上传、合并与向量化流程

## 1. 核心业务目标

- 支持大文件分片上传与断点续传
- 分片合并与一致性校验，保证对象完整性
- 语义切分与向量化可流式处理，避免大文件内存问题
- 状态可追踪、流程可恢复、接口可对接

## 2. 数据模型与状态约定

### 2.1 file_storage（文件元数据）

状态定义：
- 0 上传中（UPLOADING）
- 1 已上传/可用（UPLOADED）
- 2 已向量化（VECTORIZED）
- 3 失败/删除（FAILED）

关键字段：
- filename
- content_type
- file_size
- bucket_name
- object_name
- status
- is_public

### 2.2 file_chunks（上传分片表）

用途：
- 仅用于分片上传与合并阶段
- 合并完成后可清理

状态定义：
- 0 上传中
- 1 已上传
- 2 已合并
- 3 失败

字段建议：
- file_md5
- chunk_index
- chunk_size
- etag (分片 MD5)
- bucket_name
- object_name
- status

### 2.3 rag_chunks（建议新增，语义切分表）

用途：
- 持久化语义切分后的片段与向量
- 与上传分片完全解耦

字段建议：
- file_id
- chunk_index
- chunk_text
- chunk_tokens
- page_no / section / heading
- embedding_id
- status

## 3. Redis Key 规范

- `upload:bitmap:{user_id}:{file_md5}`: 分片完成 bitmap
- `upload:meta:{user_id}:{file_md5}`: 分片上传任务元信息
- `upload:retry:{user_id}:{file_md5}:{chunk_index}`: 分片失败重试计数

建议新增：
- `vector:task:{file_id}`: 向量化任务状态
- `vector:cursor:{file_id}`: 向量化进度游标

## 4. 接口约定（前后端对接）

### 4.1 分片上传

接口：`POST /api/v1/files/upload/chunk`
Content-Type: `multipart/form-data`

参数：
- file_md5: string
- chunk_index: int
- total_chunks: int
- chunk_size: int
- total_size: int
- file_name: string
- content_type: string
- is_public: bool
- file_chunk: file

返回：
- file_md5
- chunk_index
- uploaded: true

约定：
- chunk_index 从 0 开始
- file_md5 必须是完整文件 MD5

### 4.2 完成上传

接口：`POST /api/v1/files/upload/complete`
Content-Type: `application/json`

参数：
- file_md5
- total_chunks
- file_name
- content_type
- is_public

返回：
- file_id
- bucket_name
- object_name
- status

说明：
- 实际合并由 MQ 消费者执行，该接口用于触发与确认

### 4.3 文件列表

接口：`GET /api/v1/files`
Query:
- page
- size
- include_public (bool)

### 4.4 公共文件

接口：`GET /api/v1/files/public`
Query:
- page
- size

## 5. 核心业务流程（设计方案）

### 5.1 分片上传流程

1. 首片到达时创建 `file_storage`，状态=上传中
2. 每片记录 `file_chunks`，状态=上传中
3. 上传 MinIO 临时桶并记录分片 MD5
4. Redis bitmap 标记完成
5. 分片状态更新为已上传
6. bitmap 完整且分片数量一致时，发送 MQ 合并任务

失败处理：
- 分片失败重试达到阈值后：分片失败、file_storage 失败、meta=failed

### 5.2 合并流程（MQ 消费者）

1. 获取分片记录，按 index 排序
2. MinIO compose 合并到正式桶
3. 校验合并后的 MD5
4. 更新 file_storage 为已上传
5. 删除分片记录与临时对象
6. 清理 Redis 任务信息

失败处理：
- MD5 校验失败：file_storage=failed，清理临时分片与 Redis

### 5.3 向量化流程（新增）

触发时机：
- 合并成功后自动触发（推荐）
- 管理端可手动触发

流程：
1. 合并成功后投递向量化任务
2. 任务消费者流式读取对象，不做全量加载
3. 按页/段落解析后进行语义切分
4. 使用滑动窗口与 overlap 保证上下文连续
5. 写入向量库并落库 rag_chunks
6. file_storage.status=已向量化

## 6. 语义切分策略规范

原则：
- 上传分片仅用于传输，不参与语义切分
- 语义切分基于文档结构（页/段落/标题）

推荐策略：
- chunk_size: 512~1024 tokens
- overlap: 50~200 tokens
- 边界缓冲：上一段尾部拼接到下一段
- 保留元数据：页码、章节标题、段落序号

## 7. 失败恢复与幂等规范

- MQ 消费者处理需幂等（可重复执行）
- 失败任务可重试，游标支持断点恢复
- 以 DB 为主状态源，Redis 作为辅助

## 8. 组件职责

- API 层：参数校验、权限控制、触发任务
- Service 层：业务规则与状态更新
- MQ：解耦合并与向量化任务
- MinIO：对象存储与临时分片存储
- Redis：断点续传与临时状态

## 9. 开发规范与最佳实践

- 业务状态更新必须在 DB 中落地
- Redis 不作为最终状态判断依据
- 所有对象命名采用可重放规则（file_md5 + file_name）
- 必须记录关键日志（上传、合并、向量化）
