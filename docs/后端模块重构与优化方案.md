# 后端模块重构与优化方案

## 一、背景

NoteLLM 聊天对话系统包含三个核心模块：
- **RAG 检索模块** (`services/rag/`)：负责从知识库中检索相关文档
- **LLM 记忆模块** (`services/memory/`)：负责对话历史的存储和管理
- **向量化模块** (`services/vectorization/`)：负责文本向量化和存储

当前存在的主要问题：
1. 模块职责不清晰，部分功能重复实现
2. 代码耦合度高，难以维护和测试
3. 依赖注入混乱，存在循环依赖和初始化错误
4. 日志缺乏关键信息，难以排查问题

---

## 二、问题分析

### 2.1 RAG 检索模块问题

| 问题 | 位置 | 说明 |
|------|------|------|
| 对话历史拼接导致缓存失效 | `rag/service.py:45` | 将用户 query 与历史拼接后作为检索输入，导致缓存 key 每次不同 |
| 硬编码历史数量 | `rag/memory.py:14` | `history_limit = 8` 写死，无法配置 |
| 职责混淆 | `rag/memory.py` | `ConversationMemory` 用于 RAG 检索上下文，但实际应该只用于检索增强 |

### 2.2 LLM 记忆模块问题

| 问题 | 位置 | 说明 |
|------|------|------|
| 用户区分缺失 | `memory/markdown_memory.py:27` | `_get_object_name()` 只用 session_id，不同用户的相同 session 会混淆 |
| 功能重复 | `rag/memory.py` vs `memory/redis_memory.py` | 两个类都在读取对话历史，用途不同但易混淆 |
| 异步任务无异常处理 | `chat_message_service.py` | `asyncio.create_task()` 没有 `return_exceptions=True` |

### 2.3 向量化模块问题

| 问题 | 位置 | 说明 |
|------|------|------|
| Embedder 重复初始化 | `vector_store.py:58,110` | 构造时创建一次，`_setup_embedding_model()` 又创建一次 |
| 每次请求创建新实例 | `chat_message_service.py:126` | `MilvusVectorStore` 每次调用都新建，效率低 |

### 2.4 依赖注入问题

| 问题 | 位置 | 说明 |
|------|------|------|
| 服务层直接调用 Depends | `chat_message_service.py` | 在非路由层直接调用 FastAPI 依赖注入函数 |
| 循环依赖风险 | `dependencies/rag.py` | RAG 服务依赖注入函数不能在服务层直接调用 |

**现有依赖注入系统** (`backend/app/dependencies/`)：
- `infra.py`：Redis/MinIO/Milvus 注入 ✅ 已在 InfraProvider 中单例初始化
- `llm.py`：LLM 服务注入 ✅ 无需单例（无状态）
- `memory.py`：记忆服务注入 ✅ 良好
- `rag.py`：RAG 服务注入 ✅ 无需单例（Embedder 已通过 MilvusVectorStore 单例）

**说明**：
- LLM (ChatOpenAI) 是无状态的，每次调用独立，不需要单例
- Embedder 已通过 MilvusVectorStore 在 InfraProvider.init() 中单例初始化
- 问题在于调用方式错误（服务层直接调用 Depends），而非单例问题

## 三、改进方案

### 3.1 记忆分层架构优化

```
用户问题
    │
    ├─▶ [RAG 检索] ── 仅用当前 query 检索知识库（去掉历史拼接）
    │                   检索结果 (top-k)
    │
    └─▶ [LLM 生成]
            │
            ├─▶ 检索结果
            ├─▶ Redis 短期记忆 (最近 N 条滑动窗口)
            └─▶ MinIO 长期记忆 (Markdown)
```

**优化点：**
1. RAG 检索不再拼接对话历史，只用用户当前 query
2. 历史对话用于 LLM 生成上下文，不用于检索
3. 记忆数量可配置，去掉硬编码

### 3.2 长期记忆用户区分

```python
# 修改前
def _get_object_name(self, session_id: int) -> str:
    return f"{MEMORY_OBJECT_PREFIX}{session_id}.md"

# 修改后
def _get_object_name(self, user_id: int, session_id: int) -> str:
    return f"{MEMORY_OBJECT_PREFIX}user_{user_id}/session_{session_id}.md"
```

### 3.3 正确使用现有依赖注入系统

项目已有完善的依赖注入系统 (`app/dependencies/`)，只需要：

1. **服务层接受依赖参数**：在构造函数中接收依赖实例
2. **路由层注入依赖**：通过 `Depends()` 正确注入
3. **避免直接创建实例**：不直接在服务层 `new XXXService()`

```
路由层                    服务层
    │                        │
    ├─ Depends(get_redis) ──┤
    ├─ Depends(get_milvus)─▶ ChatMessageService(redis, milvus, ...)
    └─ Depends(get_llm) ────┘
```

### 3.4 异常处理优化

```python
# 修改前
asyncio.create_task(markdown_memory.append(...))

# 修改后
asyncio.create_task(markdown_memory.append(...), return_exceptions=True)
```
