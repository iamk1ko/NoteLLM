[遇到的问题]
我现在有一个疑问，当用户上传大文件时（例如：100MB的PDF文档）。现在的业务是会对文件进行分片上传，所有分片上传成功后再合并为整个文件存储到minio中。
但是后续我们还会对文件进行向量化时的切片，将文件按照语义信息进行切分，再转换成向量存入到向量数据库中。
但是如果把minio中的完整数据（即100MB或可能更大1GB）加载到内存再切分转成向量，是否也会遇到上传文件时的大文件问题。
但是如果在分片文件上传时，对每个分片再单独语义切分是不是会存在分片与分片直接上下文断裂的问题？如何有效的解决这个问题？有哪些常见的方法？

[解决思路]
你担心的点是对的：向量化阶段如果把完整大文件一次性加载进内存，会遇到同样的大文件问题。但不需要这么做，常见做法是“流式解析 + 分段语义切分 + 边界保护”，既避免内存爆炸，又尽量减少上下文断裂。
核心思路
- 上传分片 ≠ 语义分片。上传分片是为网络与存储可靠性服务；语义分片是为文本理解服务。两者可以完全解耦。
- 向量化阶段建议按“逻辑结构”切（页、段落、标题层级），而不是按上传分片切。
常见解决方案
1) 流式读取 / 分段解析
- PDF 逐页读取，页面级别就是天然的“稳定边界”，无需一次性加载全文件。
- 文本提取后再做语义切分（段落/标题/句子），每次只保留有限窗口在内存。
- 好处：内存占用稳定，可处理 1GB 级文件。
2) 滑动窗口 + 重叠
- 语义切分时对边界做 overlap（例如 10–20% 重叠）。
- 能缓解“上下文断裂”导致的语义缺失。
- 常见策略：按 token 长度 512–1024，overlap 50–200 token。
3) 分层切分（Hierarchical Chunking）
- 先按结构（章节→段落→句子）切，再将多个小段拼成一个 chunk。
- 记录层级元数据（章节标题/页码/段落 index）用于检索时重建上下文。
4) 跨边界桥接（Boundary Bridging）
- 处理分片边界时保留“尾部/头部缓冲区”，比如保留上一片的最后 200–500 字，拼到下一片再切分。
- 这样即使你必须按上传分片流式读取，也不会断裂。
5) 后处理拼接（Post-merge Normalize）
- 即使合并完成，向量化阶段仍然可以“逐页/逐段重切”，而非直接用上传分片。
- 上传分片仅用于存储与校验，语义切分在“合并后的对象”上进行，但不是整文件一次性读入，而是流式读取。
推荐落地策略（兼顾性能与语义）
- 合并完成后：
  - MinIO 读取对象时用 流式读取（或范围读取），避免 full read。
  - PDF 解析按页迭代，每页再做段落/句子切分 + overlap。
- 如果必须在上传分片阶段提前做向量化：
  - 加“上下文缓冲区”机制：每片处理时携带上一片尾部文本。
  - 切分时用 overlap，减少断裂。
总结
- 大文件向量化不需要一次性加载整个对象。
- 上传分片只解决“传输/存储可靠性”，语义分片应在“逻辑结构层”进行。
- 解决上下文断裂的最有效办法：滑动窗口 overlap + 结构化切分 + 边界缓冲。

[向量化业务设计]
目前已近实现了文件的分片上传和合并，接下来需要设计向量化的业务流程。
1) 当文件分片合并（位于file_merge_consumer中）完成后，发送 RabbitMQ 消息 (RABBITMQ_QUEUE_VECTORIZE_TASKS)，通知消费者(vectorize_consumer.py)进行向量化处理。
2) 消费者接收到消息后，从 MinIO 的正式桶中流式读取完整文件（避免大文件full read），进行流式解析和语义切分，生成向量并存储到向量数据库中（如 Milvus）。这部分代码写入vectorization_service.py中。
3) 向量化完成后，更新 file_storage 表中的文件状态为已向量化（2-已向量化）。

可优化的关键点
- 幂等与去重：向量化消费者必须可重复执行。建议以 file_id + version 或 file_md5 做幂等键，写入前检查 file_storage.status 与 rag_chunks 是否已完成，避免重复向量。
- 任务状态与恢复：为向量化任务维护独立状态（queued/running/success/failed）和进度游标（页号/offset），支持中断恢复与重试。
- 切分策略与上下文：不要基于上传分片切分。按页面/段落结构切，再使用滑动窗口 + overlap，避免语义断裂。
- 流式读取与内存上限：MinIO 使用 stream 或 range read；解析时用小窗口缓存，避免 full read。
- 失败重试策略：区分“可重试错误”（临时网络/模型超时）与“不可重试错误”（解析失败/格式不支持），避免无限重试。
- 一致性与事务边界：向量写入与 rag_chunks 落库建议批量提交；任务完成后再更新 file_storage.status=2。
- 观测与审计：增加关键日志与指标（处理页数/切片数/耗时/失败原因），便于排查与容量评估。
