## 会话隔离

单个用户的多次会话之间需要隔离，也就是每个聊天会话中的聊天记录不会互相影响。此外，多个用户之间的会话也需要隔离，A 用户的聊天会话和消息不会泄露/影响 B 用户的聊天会话。

## 数据库设计要点

### file_storage（文件元数据）状态规范

- `0` 上传中
- `1` 已上传/可用
- `2` 已向量化
- `3` 失败/删除

### file_chunks（上传分片表）字段建议

```
- id bigint PK
- file_id bigint（完整文件记录，合并成功后更新）
- file_md5 varchar(64)（文件唯一标识）
- chunk_index int（从 0 开始）
- chunk_size int
- etag varchar(100)（分片 MD5/ETag）
- object_name varchar(255)（分片在 MinIO 的对象名）
- bucket_name varchar(100)（临时桶）
- status tinyint（0上传中 / 1已上传 / 2已合并 / 3失败）
- create_time datetime
- update_time datetime
```

说明：
- 合并成功后删除分片记录（当前约定）
- RAG 切片表后续单独建，例如 rag_chunks

## 文件上传接口参数规范

### 1) 分片上传  
POST /api/v1/files/upload/chunk  
Content-Type: multipart/form-data

**参数**

```
- file_md5: str 文件 MD5（必填）
- chunk_index: int 分片索引，从 0 开始（必填）
- total_chunks: int 总分片数（必填）
- chunk_size: int 当前分片大小（必填）
- total_size: int 文件总大小（必填）
- file_name: str 文件名（必填）
- content_type: str MIME 类型（必填）
- is_public: bool 是否公共（管理员可用）
- file_chunk: UploadFile 分片文件本体（必填）
```

返回
- upload_id（可用 file_md5）
- chunk_index
- uploaded: true

### 2) 完成上传 & 合并  
POST /api/v1/files/upload/complete  
Content-Type: application/json
参数
- file_md5
- total_chunks
- file_name
- content_type
- is_public
返回
- file_id
- bucket_name
- object_name
- status=1（已上传）

### 3) 文件列表  
GET /api/v1/files  
Query
- page
- size
- include_public（bool，默认 true）

### 4) 公共文件  
GET /api/v1/files/public  
Query
- page
- size

## Redis Key 设计

### 1) 分片状态 Bitmap

- Key: `upload:bitmap:{user_id}:{file_md5}`
- 用法：`bit_index = chunk_index`
- 用途：判断分片是否上传完毕

### 2) 分片上传元信息 Hash

- Key: `upload:meta:{user_id}:{file_md5}`
- Fields:
  - `total_chunks`
  - `total_size`
  - `file_name`
  - `content_type`
  - `is_public`
  - `status` (uploading/merged/failed)

### 3) 临时对象名列表（可选）

- Key: `upload:parts:{user_id}:{file_md5}`
- Value: `chunk_index -> object_name`

## 文件上传业务逻辑
1. 用户通过 `POST /api/v1/files/upload/chunk` 分片上传文件（避免大文件、分片上传可解决断点上传问题）。该接口接收的参数应该包括文件本身和一些元数据（如文件名、文件类型等）。
   - file_md5: str - 文件的 MD5 值，用于唯一标识文件。
   - chunk_index: int - 当前分块的索引，从 0 开始，表示这是文件的第几个分片。
   - 文件总大小: int - 文件的总大小。
   - file_name: str - 文件名称。
   - file_chunk: (这里我不知道用什么类型) - 要上传的分片文件本身。
   - is_public: bool - 文件是否公开。
   - user_id: int - 上传用户的 ID。
2. 文件存储服务将文件保存到对象存储（本地的 MinIO 服务器中）。
   - 具体的业务代码写在 `minio_storage_service.py` 中的 `upload_file_to_minio()` 方法里。
   - 目前只给出业务逻辑就行，不需要具体代码实现。minio服务器的配置放在 `core/settings.py` 中。
   - minio 服务器的连接等相关代码写在 `minio_client.py` 中，目前我还不清楚改代码可以放在哪里。
3. 触发文件分块处理服务，将文件进行分块并存储分块信息到数据库。
   - 用户上传完整文件后（即所有分片都全部上传完毕，且分片已经能够完整合并为一个完整文件后），调用 `process_file_chunks()` 方法进行RAG分块处理。
   - RAG分块处理的具体逻辑可以参考 LangChain 的官方推荐的做法进行实现，目前只需要预留接口/或给出示例代码即可。
4. 用户可以通过 `GET /api/v1/files` 查询已上传的文件列表。

## 分片上传完整流程（你已确认的版本）

1) **首片到达时创建 `file_storage` 记录**，状态设为 `0-上传中`
2) 每个分片：
   - 插入 `file_chunks(status=0)`
   - 上传到 MinIO 临时桶
   - Redis bitmap 记录已上传分片
   - 更新分片状态 `status=1`
3) 全部分片完成后发送 MQ
4) MQ 消费者执行：
   - MinIO compose 合并到正式桶
   - 校验 `file_md5`
   - 更新 `file_storage.status=1`
   - 删除临时分片与 `file_chunks`
5) 校验失败：`file_storage.status=3` 并清理临时分片
