## 会话隔离

单个用户的多次会话之间需要隔离，也就是每个聊天会话中的聊天记录不会互相影响。此外，多个用户之间的会话也需要隔离，A 用户的聊天会话和消息不会泄露/影响 B 用户的聊天会话。

## 系统模块概览

- 用户与权限：用户隔离、管理员公共知识库
- 会话与消息：会话列表、会话详情、消息历史

- RAG 处理：分块、向量化、检索（后续实现）

## 数据库设计要点

### file_storage（文件元数据）状态规范

- `0` 上传中
- `1` 已上传/可用
- `2` 已向量化
- `3` 失败/删除

### file_chunks（上传分片表）字段建议

```
- id bigint PK
- file_id bigint（完整文件记录，合并成功后更新）
- file_md5 varchar(64)（文件唯一标识）
- chunk_index int（从 0 开始）
- chunk_size int
- etag varchar(100)（分片 MD5/ETag）
- object_name varchar(255)（分片在 MinIO 的对象名）
- bucket_name varchar(100)（临时桶）
- status tinyint（0上传中 / 1已上传 / 2已合并 / 3失败）
- create_time datetime
- update_time datetime
```

说明：
- 合并成功后删除分片记录（当前约定）
- RAG 切片表后续单独建，例如 rag_chunks

## 文件上传接口参数规范

### 1) 分片上传  
POST /api/v1/files/upload/chunk  
Content-Type: multipart/form-data

**参数**

```
- file_md5: str 文件 MD5（必填）
- chunk_index: int 分片索引，从 0 开始（必填）
- total_chunks: int 总分片数（必填）
- chunk_size: int 当前分片大小（必填）
- total_size: int 文件总大小（必填）
- file_name: str 文件名（必填）
- content_type: str MIME 类型（必填）
- is_public: bool 是否公共（管理员可用）
- file_chunk: UploadFile 分片文件本体（必填）
```

- chunk_index
- uploaded: true

Content-Type: application/json
参数

- file_name
- content_type
- is_public
返回
- file_id
- bucket_name
- object_name
- status=1（已上传）

### 4) 公共文件

## Redis Key 设计

### 1) 分片状态 Bitmap

- Key: `upload:bitmap:{user_id}:{file_md5}`
- 用法：`bit_index = chunk_index`
- 用途：判断分片是否上传完毕

### 2) 分片上传元信息 Hash

- Key: `upload:meta:{user_id}:{file_md5}`
- Fields:
  - `total_chunks`
  - `total_size`
  - `file_name`
  - `content_type`
  - `is_public`
  - `status` (uploading/merged/failed)

### 3) 临时对象名列表（可选）

- Key: `upload:parts:{user_id}:{file_md5}`
- Value: `chunk_index -> object_name`

## RabbitMQ 消费者脚本（示例逻辑）

- 消费者仅展示处理逻辑，不包含完整生产实现

## 后端核心业务设计与接口约定

说明：
- 该章节用于“开发规范 + 接口约定 + 核心业务流程”统一说明
- 供后端实现与前后端对接使用

### 1. 核心业务目标

- 支持大文件分片上传与断点续传
- 分片合并与一致性校验，保证对象完整性
- 语义切分与向量化可流式处理，避免大文件内存问题
- 状态可追踪、流程可恢复、接口可对接

### 2. 数据模型与状态约定

file_storage 状态：
- 0 上传中（UPLOADING）
- 1 已上传/可用（UPLOADED）
- 2 已向量化（VECTORIZED）
- 3 失败/删除（FAILED）

file_chunks 状态：
- 0 上传中
- 1 已上传
- 2 已合并
- 3 失败

rag_chunks（建议新增）：
- 与上传分片解耦，用于语义切分与向量结果持久化

### 3. Redis Key 规范

- `upload:bitmap:{user_id}:{file_md5}`: 分片完成 bitmap
- `upload:meta:{user_id}:{file_md5}`: 分片上传任务元信息
- `upload:retry:{user_id}:{file_md5}:{chunk_index}`: 分片失败重试计数

建议新增：
- `vector:task:{file_id}`: 向量化任务状态
- `vector:cursor:{file_id}`: 向量化进度游标

### 4. 接口约定（前后端对接）

分片上传：`POST /api/v1/files/upload/chunk`
- Content-Type: multipart/form-data
- 参数：file_md5, chunk_index, total_chunks, chunk_size, total_size, file_name, content_type, is_public, file_chunk
- 返回：file_md5, chunk_index, uploaded

完成上传：`POST /api/v1/files/upload/complete`
- Content-Type: application/json
- 参数：file_md5, total_chunks, file_name, content_type, is_public
- 返回：file_id, bucket_name, object_name, status

文件列表：`GET /api/v1/files`
- Query：page, size, include_public

公共文件：`GET /api/v1/files/public`
- Query：page, size

### 5. 核心业务流程（设计方案）

分片上传：
1) 首片到达时创建 `file_storage(status=0)`
2) 每片写入 `file_chunks(status=0)` 并上传临时桶
3) Redis bitmap 标记完成，更新 `file_chunks(status=1)`
4) bitmap 完整且分片数量一致时发送 MQ 合并任务

合并流程：
1) 拉取分片并按 index 排序
2) MinIO compose 合并到正式桶
3) 校验 `file_md5`
4) 更新 `file_storage.status=1`
5) 删除临时分片与 `file_chunks`
6) 清理 Redis bitmap/meta

向量化流程（新增）：
1) 合并成功后投递向量化任务
2) 流式读取对象并按结构切分（页/段落/标题）
3) 使用滑动窗口与 overlap 保证上下文连续
4) 向量写入向量库并落库 rag_chunks
5) 更新 `file_storage.status=2`

### 6. 语义切分策略规范

- 不使用上传分片做语义切分
- chunk_size: 512~1024 tokens
- overlap: 50~200 tokens
- 保留页码/章节/段落元数据

### 7. 失败恢复与幂等规范

- MQ 消费者需幂等处理
- 失败任务可重试，游标支持断点恢复
- DB 为主状态源，Redis 为辅助

### 8. 组件职责

- API 层：参数校验、权限控制、触发任务
- Service 层：业务规则与状态更新
- MQ：解耦合并与向量化任务
- MinIO：对象存储与临时分片存储
- Redis：断点续传与临时状态

### 9. 开发规范与最佳实践

- 所有状态更新必须落库
- Redis 不作为最终状态判断依据
- 对象命名规则需确定性（file_md5 + file_name）
- 关键流程必须有日志

### 10. 向量化任务设计（新增）

触发时机：
- `file_merge_consumer` 合并成功后发送向量化任务消息

消息队列：
- `RABBITMQ_QUEUE_VECTORIZE_TASKS`

消息体建议字段：
- file_id
- file_md5
- bucket_name
- object_name
- content_type
- user_id

消息体约束与示例：
- 字段均为必填，确保消费者幂等与定位对象
- file_md5 作为幂等键，用于跳过已完成向量化任务

示例：
```
{
  "file_id": 1001,
  "file_md5": "7f9c2ba4e88f9a...",
  "bucket_name": "upload-final",
  "object_name": "7f9c2ba4e88f9a.../demo.pdf",
  "content_type": "application/pdf",
  "user_id": 12
}
```

任务状态与游标（Redis）：
- `vector:task:{file_md5}`: queued/running/success/failed
- `vector:cursor:{file_md5}`: page_no / offset

向量化处理流程：
1) 任务消费者接收消息后进行幂等检查（file_md5）
2) MinIO 正式桶流式读取对象，采用 PDF 页级解析
3) 使用滑动窗口 + overlap 切分文本
4) 生成向量并写入向量数据库（批量提交）
5) 写入 rag_chunks 元数据
6) 更新 file_storage.status=2

解析器策略选择：
- 当前优先 PDF 页级解析
- 后续如需多格式支持，扩展为统一抽象解析器接口

推荐切分参数：
- chunk_size: 512~1024 tokens
- overlap: 50~200 tokens

rag_chunks 与向量库字段映射：
- rag_chunks.file_id -> 向量库 partition 或 metadata.file_id
- rag_chunks.chunk_index -> 向量库 metadata.chunk_index
- rag_chunks.chunk_text -> 向量库 text 字段或 metadata.text
- rag_chunks.chunk_tokens -> 向量库 metadata.tokens
- rag_chunks.page_no -> 向量库 metadata.page_no
- rag_chunks.section -> 向量库 metadata.section
- rag_chunks.embedding_id -> 向量库主键或外部引用
